## 주요 논의 아이디어

### 1. 냉장고 연동 레시피 앱 (집중 논의)

**핵심 기능:**

- 냉장고 재료 인식 (카메라/음성)
- AI 기반 레시피 생성
- 유통기한 관리 및 알림
- 외식 메뉴 월드컵
- 커뮤니티 기능 (후기 작성)

**부가 기능:**

- 재료별 가능한 레시피 개수 표시
- 알레르기 정보 입력
- 북마크 기능
- 쿠팡 파트너스 연동

**기술적 구현 방안:**

- **온디바이스 AI**: 갤럭시 S23 기준 경량화 모델 활용
- **영수증 OCR**: 자동 재료 등록
- **파인튜닝**: 레시피 데이터셋 학습
- **음성 가이드**: 요리 중 음성으로 다음 단계 안내

**유통기한 관리의 난점:**

- 사용자 입력 부담 증가
- 계절별/보관 방법별 차이
- 통상적 기준 vs 실제 유통기한 차이

**해결 방안:**

- 영수증 OCR로 구매일 기준 자동 설정
- 통상적 유통기한을 디폴트로 제공 후 사용자가 수정
- 계절별 일괄 조정 기능

### 2. 장애인 접근성 앱 (계단/경사로 감지)

**기술적 접근:**

- **센서 기반**: 핸드폰 가속도/자이로 센서로 계단 감지
- **사진 기반**: 사용자가 직접 계단/경사로 촬영 및 제보
- **하이브리드**: 센서 + 사진 병행

**기존 서비스 조사:**

- 휠체어 내비게이션 앱 이미 존재
- 해외: 전 세계 휠체어 접근성 정보 제공 앱

**우려사항:**

- 에스컬레이터를 경사로로 오인할 가능성
- 계단+경사로 혼재 시 판단 어려움
- **데이터 수집과 앱 주목적 간 연결성**

### 3. 약속 관리 앱 + 접근성 데이터 수집

**기본 구상:**

- **메인 기능**: 대학생 팀플/동아리 약속 잡기 앱
- **서브 기능**: 이동 중 접근성 데이터 자동 수집
- **목적**: 장애인도 비장애인도 함께 이용 가능한 약속 앱

**초기 아이디어 의도:**

> "장애인 다리가 아프신 분들이 그냥 그 바퀴로 바퀴가 다리인 거잖아.
> 

우리는 두 다리로 걸을 수도 있고 계단도 오를 수도 있잖아. 

우리가 일상적으로 가는 약속 장소를 약간 장애인 분들도 약간 편하게 갔으면 좋겠다"

**기능 구성:**

- 팀플/동아리 모임 일정 조율
- 장소 추천 및 예약
- 걸어가는 동안 센서로 계단 감지
- GPS 위치와 연동한 접근성 지도 구축

**주요 우려사항:**

1. **앱 정체성 혼재**: "약속 앱인 거야? 데이터 수집하는 앱인 거야?"
2. **사용자 경험의 부자연스러움**: 갑자기 뜬금없는 데이터 수집 기능
3. **프라이버시 우려**: GPS 위치 추적에 대한 거부감
4. **기능 간 연결성 부족**: 두 기능이 자연스럽게 어우러지지 않음

**구체적 사용 시나리오 우려:**

> "우리 팀 팀플 하러 가야지 야 우리끼리 이거 잡자 이렇게 잡았어.
 근데 걸어가는 동안 데이터를 수집해도 괜찮을까요? 뭐 이런 거 한번 묻긴 할 거 아니요?"
> 

**대안적 접근:**

- 봉사 목적 명시 ("걸어서 지구를 깨끗하게 만들어 봐요")
- 특정 사용자층 타겟팅 (부모-자녀, 친구 간)
- 선택적 기능 (만보기 형태)

## 세부 기술 검토

### AI/ML 모델 관련

- **경량화 모델**: 100MB 이하로 온디바이스 구현 가능
- **파인튜닝**: 13만 개 데이터로 100번 학습 시 100MB 이하
- **이미지 인식**: EfficientNet, YOLO 등 경량 모델 활용

### 데이터 처리

- **레시피 데이터셋**: 만 개 이상의 레시피 데이터 확보 필요
- **재료 이미지**: 구매 또는 직접 생성 (약 10만원 예상)
- **OCR 정확도**: 캡처 이미지가 사진보다 인식률 높음

### API 비용

- **Gemini API**: 무료 사용 우선 검토
- **토큰 비용**: 프로젝트 기간 중 10만원 내외 예상

### 냉장고 관리 앱

- **AI 냉장고 레시피 앱**: 유사 기능이지만 사용자 수 적음
- **차별화 포인트**: 음성 가이드, 커뮤니티, 쇼핑 연동

### 접근성 앱

- **휠체어 내비게이션**: 이미 상용화된 서비스들 존재
- **차별화 포인트**: 센서 기반 자동 데이터 수집

## 핵심 딜레마와 팀 논의

### 앱 정체성에 대한 근본적 질문

**핵심 쟁점**: "약속 앱을 만드는데 데이터를 넣을 건지 아니면은 데이터를 제공하고 싶어서 약속 앱을 만드는 건지"

### 볼륨 확장에 대한 고민

**AI 기능 부족 우려**: "여기서 이걸로만 하면 볼륨이 안 나오니까는 AI도 어떻게 낼 거야?"

**추가 가능한 AI 기능들:**

- 약속 장소 추천 AI
- 경로 예측 AI
- 참석자 특성 기반 메뉴/장소 추천

### 기존 서비스와의 경쟁

**발견된 경쟁 앱들:**

- 휠체어 내비게이션 앱들 이미 존재
- 해외: 전 세계 휠체어 접근성 정보 제공 앱
- 국내: 네이버 지도에 일부 시설 정보 제공

### 팀원들의 심리적 부담

**완벽주의적 접근**: "우리가 어차피 이거를 컨설턴트 님을 설득을 시키고... 결과적으로는 다른 사람들 평가가 들어갈 것 같은데 우리 게 그럼 이거 왜 쓰는 걸까라고 생각이 들면 난 그게 좀 아쉽다"

**자기 검증의 중요성**: "우리끼리 엄청난 비판을 했을 때 대답이 되면 그러면 그거는 난 가능성이 있는 거라고 생각을 해서"

### 1. 사용자 편의성 vs 기능 복잡성

**문제**: 기능이 많아질수록 사용자 입력 부담 증가
**해결책**:

- MVP(Minimum Viable Product) 단계별 구현
- 자동화 기능 최대한 활용
- 사용자 선택권 확대

### 2. 앱 정체성의 명확성

**문제**: 여러 목적이 섞여 앱의 핵심 가치 흐림
**해결책**:

- 명확한 주기능과 부기능 구분
- 사용자에게 부기능의 가치 명확히 전달

### 3. 기술적 실현 가능성

**문제**: 센서 기반 계단 감지의 정확도 불확실
**해결책**:

- 사전 테스트를 통한 검증
- 대안 기술 준비 (사진 기반)
